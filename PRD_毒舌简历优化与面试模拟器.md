# PRD：毒舌简历优化与面试模拟器

## 1. 文档信息

### 1.1 版本信息
- 版本：V3
- 更新时间：2026-04-13
- 当前阶段：本地前端 Demo 已完成，进入 MVP 细化与真实 AI 接入规划阶段

### 1.2 产品名称
- 中文暂定名：简历毒打官
- 英文暂定名：Roast My Resume

### 1.3 一句话定位
一款面向年轻求职者的 AI 求职产品，分为“简历生成器”和“简历诊断与模拟面试”两大功能模块：先帮用户快速生成一份能投的简历，再用毒舌但专业的前大厂 HR 视角诊断、重写、追问，帮助用户把简历和面试一起打磨到可投递状态。

## 2. 产品概述

### 2.1 产品背景
市面上大多数简历工具存在四类问题：

1. 建议太温和，用户看完仍然不知道自己到底差在哪。
2. 输出太模板化，只会说“建议量化成果”，却不会真正帮用户改写。
3. 简历优化和面试准备割裂，用户改完简历后依旧不知道会被怎么问。
4. 结果交付感不足，用户拿到的是建议，而不是一份可以继续编辑和导出的正式简历。

年轻求职者尤其是应届生、校招生、1-5 年经验的职场人，更愿意为“风格鲜明、反馈直接、结果可用”的产品停留和付费。

### 2.2 产品核心价值
本产品围绕求职动作拆成两个独立但可互通的模块：

1. 简历生成器：用户直接填写信息，AI 优化内容并生成正式简历模板，支持连续修改、上传照片、导出 PDF/图片。
2. 简历诊断与模拟面试：用户上传或粘贴已有简历，输入目标 JD，AI 进行毒舌点评、改写关键片段，并生成面试追问。

### 2.3 产品形态
首个版本为本地可运行的 Web 前端 Demo，后续演进为在线产品。

当前 Demo 已覆盖：
1. 双模块工作台
2. 多模板简历生成
3. 本地内容优化
4. 简历诊断与重写
5. 面试追问预览
6. 网页内继续编辑
7. PDF / PNG 导出

## 3. 产品目标

### 3.1 业务目标
1. 建立高记忆点的求职工具心智，形成差异化传播。
2. 提升从“输入信息”到“拿到结果”的完成率。
3. 跑通“简历生成 -> 简历优化 -> 面试准备 -> 付费升级”的基础漏斗。
4. 为后续接入模型 API、账户系统、历史记录和在线导出能力打基础。

### 3.2 用户目标
1. 快速知道自己的简历哪里弱，而不是收到空泛建议。
2. 拿到可以直接继续修改和导出的简历成品。
3. 让简历表达更贴近目标 JD。
4. 提前暴露面试风险点，降低面试翻车概率。
5. 减少“不会写、不会改、不会答”的心理负担。

### 3.3 北极星指标
- 完成“生成或上传简历 -> 获得优化结果 -> 导出正式简历”的用户占比

## 4. 目标用户

### 4.1 核心用户
1. 18-30 岁年轻求职者
2. 应届生 / 校招生
3. 1-5 年经验、准备跳槽进入更高平台的人群
4. 互联网、产品、运营、市场、数据、设计、前端等知识型岗位求职者

### 4.2 典型用户画像

**用户 A：应届生产品/运营候选人**
- 痛点：经历不算少，但表达像课程总结和流水账。
- 需求：需要有人直接指出“你这段为什么没说服力”，并帮忙改成能投的版本。

**用户 B：2-3 年经验跳槽用户**
- 痛点：做过事，但不擅长提炼结果、指标和个人贡献。
- 需求：围绕目标 JD 做定向重写，而不是泛化优化。

**用户 C：转岗用户**
- 痛点：经历是真实的，但关键词和岗位语言不匹配，容易被筛掉。
- 需求：让已有经历被重新组织成“适合这个岗位”的表达。

## 5. 核心痛点分析

### 5.1 显性痛点
1. 简历投出去没有面试，不知道问题在哪。
2. 会做事，但不会把经历写得像成果。
3. 看过很多模板，依然不知道自己的内容该怎么组织。
4. 改简历时容易越改越虚，担心“像造假”。
5. 面试中经常被追问到答不上来。

### 5.2 隐性痛点
1. 用户想要的不是建议，而是“帮我先做出第一版可用结果”。
2. 用户需要的是岗位语言翻译能力，而不是写作鸡汤。
3. 用户希望结果有交付感，最好能直接变成一份正式简历。
4. 用户愿意接受毒舌，但前提是毒舌必须专业、有帮助、有边界。

## 6. 产品定位与差异化

### 6.1 产品定位
年轻人的 AI 毒舌 HR 陪练官。

### 6.2 差异化卖点
1. 先骂后改：不是温柔提醒，而是高冲击感反馈。
2. 风格鲜明：前大厂 HR 人设，形成产品记忆点。
3. 批评后立刻给方案：每条问题后都配修改建议或优化版文案。
4. 围绕 JD 做定向优化，而不是泛泛重写。
5. 从简历到面试形成闭环，不只停在文案层。
6. 最终交付的是“可继续编辑并导出的正式简历”。

### 6.3 风格边界
1. 只批评简历内容，不攻击用户人格。
2. 不涉及学历歧视、地域歧视、外貌攻击、性别偏见等内容。
3. 不鼓励造假，不虚构经历。
4. 所有毒舌都必须服务于结果提升，而不是为了冒犯。

## 7. 产品结构

### 7.1 总体结构
产品分为两个一级功能：

1. 功能一：简历生成器
2. 功能二：简历诊断与模拟面试

两个模块相互独立，但结果可互通。

### 7.2 模块关系
1. 用户可以直接从零开始，用“简历生成器”填写信息并生成简历。
2. 用户也可以带着已有简历进入“简历诊断与模拟面试”。
3. 诊断模块输出的优化结果可以回流到简历生成器，继续整理为正式简历。
4. 简历生成器产出的简历后续也可以再次进入诊断模块进行打磨。

## 8. 核心使用场景

### 8.1 场景一：从零生成简历
用户没有现成简历，或现有简历太乱，希望直接填写信息并快速得到一份结构完整、排版正式、可继续编辑的简历。

### 8.2 场景二：已有简历优化
用户已经有简历，但表达一般，希望通过 AI 的毒舌诊断快速定位问题并改写关键片段。

### 8.3 场景三：面向目标 JD 定向优化
用户手里有具体职位，希望简历内容和岗位要求更贴近。

### 8.4 场景四：改完简历继续练面试
用户希望知道“这个简历最容易被怎么追问”，提前准备回答。

### 8.5 场景五：导出正式结果
用户希望不是拿到一堆建议，而是直接拿到一份能编辑、能导出、能投递的正式文件。

## 9. 用户流程

### 9.1 功能一：简历生成器流程
1. 用户进入“简历生成器”。
2. 选择简历模板。
3. 填写姓名、目标岗位、电话、邮箱、城市、教育背景、照片等基础信息。
4. 填写个人概述、核心经历、项目经历、技能等内容。
5. 点击“AI 一键优化内容”。
6. 系统生成优化后的简历草稿。
7. 用户在网页右侧预览区继续直接修改。
8. 用户切换模板查看不同版式效果。
9. 用户导出为 PDF 或 PNG 图片。

### 9.2 功能二：简历诊断与模拟面试流程
1. 用户进入“简历诊断与模拟面试”。
2. 上传 PDF 简历或粘贴简历文本。
3. 输入目标岗位 JD。
4. 点击“开始 HR 诊断”。
5. 系统输出总评、问题分类、评分和关键吐槽。
6. 用户点击“一键重写”获得优化版简历片段。
7. 系统生成模拟面试追问。
8. 用户可将优化结果带回“简历生成器”继续整理成正式简历。

### 9.3 当前 Demo 已验证流程
1. 双模块切换
2. 本地规则内容优化
3. 模板切换
4. 网页内连续编辑
5. PDF / PNG 导出
6. 诊断结果回流生成器

## 11. 功能详细设计

### 11.1 简历生成器

**目标**
帮助用户从零生成一份正式、可编辑、可导出的简历，而不是只给出文字建议。

**当前能力**
1. 支持填写基础信息、经历、项目、技能。
2. 支持上传个人照片。
3. 支持多模板切换。
4. 支持网页内直接编辑预览内容。
5. 支持导出 PDF 和 PNG。
6. 支持本地规则优化内容。

**后续增强方向**
1. 将“经历/项目”从大文本框升级为可增删卡片表单。
2. 支持排序、拖拽、隐藏字段。
3. 支持不同模板的专属字段。
4. 支持真正固定 A4 分页导出。

### 11.2 简历模板系统

**目标**
模板不是简单换色，而是让不同用户场景有真正不同的版式结构和信息层级。

**模板设计原则**
1. 每套模板必须在结构上明显不同。
2. 模板应匹配不同岗位和用户类型。
3. 模板既要美观，也要适合招聘者阅读。
4. 模板必须兼容网页内连续编辑与导出。

**当前模板方向**
1. Sharp：锋利型，适合年轻用户和增长、运营、产品方向。
2. Calm：专业型，适合稳重通用岗位。
3. Mono：极简型，适合开发、数据等偏理性岗位。
4. Magazine：杂志型，适合品牌、内容、市场方向。
5. Studio：作品集型，适合设计、创意、内容岗位。
6. Executive：高管型，适合资深岗位与商务风格。
7. Campus：校招型，适合应届生、实习生。
8. Neo：未来型，适合年轻化、科技感表达。

**后续模板规划**
1. 数据分析版
2. 咨询顾问版
3. 海外求职版
4. 双语简历版
5. 特定大厂风格版

### 11.3 照片上传

**目标**
满足用户对“带头像简历”的场景需求，尤其适用于校招、创意类岗位和个人品牌型表达。

**当前能力**
1. 支持上传本地图片。
2. 在部分模板中展示头像。
3. 支持头像预览。

**设计要求**
1. 图片应自动裁切为适合版式的比例。
2. 失败时需要有兜底提示。
3. 导出时头像应稳定保留。

### 11.4 AI 一键优化内容

**目标**
根据用户填写的信息自动优化个人概述、经历、项目和技能表达。

**当前状态**
1. Demo 版采用本地规则引擎实现基础润色。
2. 可演示完整交互链路，但表达仍偏模板化。

**正式版本能力要求**
1. 通过模型 API 实现自然语言改写。
2. 支持按岗位方向做风格化优化。
3. 严格遵守“优化表达，不虚构经历”的边界。
4. 给出“优化前 / 优化后 / 建议补充项”的结构化输出。

**Prompt 方向**
1. 保留真实经历，不得捏造未提供的数据和成绩。
2. 优先强化动作、目标、结果、协作、个人贡献。
3. 表达应符合简历 bullet 语言，而不是口语化长段落。
4. 优先吸收 JD 关键词。

### 11.5 简历输入与 PDF 上传

**目标**
降低用户进入诊断流程的成本。

**当前能力**
1. 支持上传 PDF 入口。
2. 支持粘贴简历文本。
3. 当前 Demo 中 PDF 主要用于流程展示，未完成真实解析。

**正式版本能力要求**
1. 解析 PDF 正文。
2. 自动识别教育、经历、项目、技能等结构。
3. 解析失败时允许用户手动修正。

### 11.6 JD 输入与解析

**目标**
让诊断和改写围绕目标岗位进行，而不是泛泛优化。

**系统处理逻辑**
1. 提取职责关键词。
2. 提取能力关键词。
3. 提取结果导向词。
4. 对比简历覆盖情况。

**输出内容**
1. 匹配度评分。
2. 命中关键词。
3. 缺失关键词。
4. 诊断中“与 JD 不匹配”的具体原因。

### 11.7 毒舌诊断

**目标**
让用户快速意识到简历真正的问题，并对修改产生行动意愿。

**输出结构**
1. 一句总评
2. 三个核心评分
3. 多条毒舌点评
4. 对应修改建议

**问题分类**
1. 流水账
2. 缺结果
3. 缺数据
4. 缺个人贡献
5. 关键词不贴 JD
6. 技能表达过虚

### 11.8 一键重写

**目标**
把“知道有问题”转化为“拿到能用的优化版本”。

**输出结构**
1. 原始表达
2. 问题说明
3. 优化版表达

**重写原则**
1. 不虚构数据与成果。
2. 优先重写最危险片段。
3. 表达必须更适合简历投递场景。
4. 可继续带入简历生成器做最终整理。

### 11.9 模拟面试

**目标**
让用户知道这份简历最容易在哪些点被追问。

**当前能力**
1. 生成 3-5 条追问。
2. 覆盖结果质疑、贡献质疑、数据口径质疑、岗位匹配质疑。

**后续规划**
1. 支持用户逐题作答。
2. AI 根据回答继续追问。
3. 输出示范回答。
4. 输出面试复盘报告。

### 11.10 导出系统

**目标**
让用户从产品中直接拿走正式交付结果。

**当前能力**
1. 通过浏览器打印导出 PDF。
2. 支持导出 PNG 图片。

**后续规划**
1. 固定 A4 排版。
2. 多页分页控制。
3. 高清 JPG / PNG 输出。
4. 模板级导出样式控制。

## 13. 体验原则

### 13.1 产品体验原则
1. 反馈必须即时可感知。
2. 批评必须配方案。
3. 结果必须可编辑，而不是只读文本。
4. 风格要有攻击性包装，但交付要有专业可信度。
5. 页面要像“求职工作台”，不是普通后台表单。

### 13.2 语气原则
1. 尖锐但专业。
2. 有记忆点，但不为段子牺牲信息密度。
3. 不给空建议，不做无意义夸奖。

### 13.3 视觉原则
1. 避免统一模板换色的假丰富。
2. 模板应有明确版式差异。
3. 兼顾桌面端和移动端浏览。
4. 保证打印与导出场景下的可读性。

## 15. 内容安全与规则约束

### 15.1 内容安全规则
1. 禁止对用户人格进行攻击。
2. 禁止输出歧视性内容。
3. 禁止鼓励用户伪造简历经历。
4. 禁止输出违法违规建议。

### 15.2 真实性约束
1. 默认只优化表达，不杜撰事实。
2. 对缺失的数据只提示“建议补充”，不直接编造。
3. 如果用户没有给出明确成果，输出中应保留提示位。

### 15.3 风格约束
1. 毒舌必须针对内容，不针对人。
2. 每条批评必须附带修改方向。
3. 高风险内容应自动降级为更稳妥语气。

## 20. 风险与应对

### 20.1 风格风险
风险：毒舌风格容易越界，造成冒犯或舆情问题。  
应对：建立风格模板和安全策略，确保“刀法准”但“不伤人”。

### 20.2 真实性风险
风险：用户可能试图把产品当成包装甚至造假工具。  
应对：明确“优化表达，不伪造经历”，对缺失数据仅给提示位。

### 20.3 模板同质化风险
风险：如果模板只是换色，用户会觉得产品很浅。  
应对：模板设计必须做真实结构差异，并按岗位场景拆分。

### 20.4 结果可信度风险
风险：如果 AI 改写过于模板化，用户会觉得像文案生成器。  
应对：强化 JD 匹配、真实经历抽取、结构化改写和人工可编辑性。

### 20.5 导出体验风险
风险：网页预览好看，但 PDF 导出翻车。  
应对：尽快推进固定 A4 排版和模板级导出样式控制。
